# Migration Workflow

## Purpose

Alembic owns database schema evolution for the modular monolith.

Database schema changes must remain aligned with bounded-context ownership and the architectural principles of the system.

This document defines the workflow for creating, reviewing, and applying database migrations.

---

# Architecture Principles

## Single Migration Stream

The platform currently operates as a modular monolith.

A single Alembic migration stream is maintained for the entire application.

Future service extraction may introduce independent migration streams, but this is not currently supported.

---

## Bounded Context Ownership

Each database table must be owned by exactly one bounded context.

Example:

Identity Context

* users
* credentials
* auth_sessions
* auth_challenges

Future contexts should own their own tables.

Cross-context access should occur through application services, events, repositories, or read models rather than shared table ownership.

---

## Schema Ownership Rules

A bounded context:

* Owns its tables
* Owns its indexes
* Owns its constraints
* Owns its migrations

Other contexts must not modify another context's tables without an explicit architectural decision.

---

# Configuration

Alembic reads `DATABASE_URL` from the environment.

Example:

```text
postgresql://platform:platform@postgres:5432/platform
```

The Alembic environment converts this URL to SQLAlchemy's async PostgreSQL driver form:

```text
postgresql+asyncpg://platform:platform@postgres:5432/platform
```

---

# Creating Migrations

After modifying SQLAlchemy models:

```shell
uv run alembic revision --autogenerate -m "describe change"
```

Example:

```shell
uv run alembic revision --autogenerate -m "add auth session table"
```

---

# Reviewing Migrations

Autogenerated migrations must always be reviewed before execution.

Review for:

* Correct table ownership
* Correct column definitions
* Correct indexes
* Correct foreign keys
* Correct constraints
* Unintended destructive changes

Never apply autogenerated migrations without review.

---

# Applying Migrations

Apply the latest migration:

```shell
uv run alembic upgrade head
```

Using Docker Compose:

```shell
docker compose run --rm api alembic upgrade head
```

---

# Inspecting Migration State

Current revision:

```shell
uv run alembic current
```

Migration history:

```shell
uv run alembic history
```

---

# Rolling Back Migrations

Rollback one migration:

```shell
uv run alembic downgrade -1
```

Rollback to a specific revision:

```shell
uv run alembic downgrade <revision>
```

---

# Introducing New Contexts

When a new bounded context is introduced:

1. Create context-owned SQLAlchemy models.
2. Register model metadata in Alembic.
3. Generate migrations.
4. Verify table ownership.
5. Review migration output.
6. Apply migration.

---

# Alembic Metadata Registration

When new context model modules are added, ensure they are imported in:

```text
alembic/env.py
```

This allows Alembic autogenerate to discover the metadata.

---

# Future Evolution

Current State

* Modular monolith
* Single database
* Single migration stream

Future Possibilities

* CQRS read databases
* Outbox tables
* Projection tables
* Independent service databases
* Service-specific migration streams

Until service extraction occurs, all schema evolution remains managed through the single Alembic migration workflow.
